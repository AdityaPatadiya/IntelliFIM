# Postgres Migration (v2-1) — design

**First v2 sub-project.** Replaces SQLite in 3 services (auth-backend, response-orchestrator, reporting) with a single Postgres instance hosting 3 isolated databases. Foundational v2 work — every subsequent v2 sub-project that touches persistence (HA, TLS, pgaudit, alembic) builds on this.

**Date:** 2026-05-23
**Author:** IntelliFIM team
**Status:** Approved — ready for implementation plan.

---

## 1. Goal

Migrate the 3 v1 SQLite-backed services to Postgres:
1. `auth-backend` (`/data/users.db` → `postgresql://auth:***@postgres:5432/auth_backend`)
2. `response-orchestrator` (`/data/approvals.db` → `postgresql://orchestrator:***@postgres:5432/orchestrator`)
3. `reporting` (`/data/reporting.db` → `postgresql://reporting:***@postgres:5432/reporting`; PDFs stay on filesystem volume)

Each service keeps its existing class shape (`UsersStore`, `ApprovalStore`, `ReportingStore`) and method signatures intact — only the transport (`aiosqlite` → `asyncpg`) changes. API surfaces above the stores are untouched. No new behaviors.

Establishes the Postgres bootstrap pattern (init script + per-service users + databases) + the `testcontainers-python` test infrastructure pattern that every future v2 sub-project touching persistence inherits.

## 2. Architecture

**1 new Compose service: `postgres`.**

| Field | Value |
|---|---|
| Image | `postgres:16-alpine` |
| Port | `127.0.0.1:5432:5432` |
| Network | `bus` (same as the 6 in-house services) |
| Volume | `postgres_data` (new named volume) |
| Init script | `data-plane/postgres/init.sh` mounted at `/docker-entrypoint-initdb.d/01-init.sh` |
| Healthcheck | `pg_isready -U postgres -d postgres` |

**Stack count: 27 → 28 services** in normal `docker compose up -d`.

**Isolation model: one Postgres, 3 separate databases.**

| DB | Owner user | Service connects with |
|---|---|---|
| `auth_backend` | `auth` | `postgresql://auth:${POSTGRES_AUTH_PASSWORD}@postgres:5432/auth_backend` |
| `orchestrator` | `orchestrator` | `postgresql://orchestrator:${POSTGRES_ORCH_PASSWORD}@postgres:5432/orchestrator` |
| `reporting` | `reporting` | `postgresql://reporting:${POSTGRES_REPORTING_PASSWORD}@postgres:5432/reporting` |
| `postgres` (admin) | `postgres` (superuser) | `${POSTGRES_ROOT_PASSWORD}` — used only by the init script |

Each service user OWNS its database (full DDL — needed because `init_schema()` runs `CREATE TABLE IF NOT EXISTS` at first boot). v3 production hardening will replace owner-rights with a separate ddl-only migration user + service users with row-level rights only (DML).

**Driver: `asyncpg` (raw SQL, no ORM).** Each store uses an `asyncpg.create_pool(...)` (minsize=1, maxsize=8). Removes v1's `asyncio.Lock` since Postgres handles concurrency natively. All 20 SQL statements across the 3 stores are mechanically translated from aiosqlite `?` placeholders to asyncpg `$1, $2, ...`.

**Schema management: hand-rolled `init_schema()`** — same pattern as v1. Each service's startup calls `await store.init_schema()` which runs idempotent `CREATE TABLE IF NOT EXISTS ...` + `CREATE INDEX IF NOT EXISTS ...` against its database. No alembic in v2; deferred to v3.

**Data migration: fresh start, no SQLite → Postgres data copy.** auth-backend re-seeds the admin user on first boot (same v1 behavior). Orchestrator + reporting start with empty tables. v3 production migrations will need a one-shot SQLite → Postgres script; v2 explicitly skips it.

**Test infrastructure: `testcontainers-python`.** Each service's `[dev]` extras add `testcontainers[postgres]`. A session-scoped pytest fixture starts a Postgres container at test-session start; per-test fixtures create randomly-named databases for hermetic isolation.

## 4. Postgres Service Bootstrap

### 4.1 Compose service block

```yaml
  postgres:
    image: postgres:16-alpine
    container_name: postgres
    networks: [bus]
    ports:
      - "127.0.0.1:5432:5432"
    environment:
      POSTGRES_USER: "postgres"
      POSTGRES_PASSWORD: "${POSTGRES_ROOT_PASSWORD}"
      POSTGRES_DB: "postgres"
      POSTGRES_AUTH_PASSWORD: "${POSTGRES_AUTH_PASSWORD}"
      POSTGRES_ORCH_PASSWORD: "${POSTGRES_ORCH_PASSWORD}"
      POSTGRES_REPORTING_PASSWORD: "${POSTGRES_REPORTING_PASSWORD}"
    volumes:
      - postgres_data:/var/lib/postgresql/data
      - ./postgres/init.sh:/docker-entrypoint-initdb.d/01-init.sh:ro
    healthcheck:
      test: ["CMD-SHELL", "pg_isready -U postgres -d postgres"]
      interval: 5s
      timeout: 5s
      retries: 10
```

### 4.2 Init script `data-plane/postgres/init.sh`

`postgres:16-alpine`'s entrypoint runs `*.sh` and `*.sql` files in `/docker-entrypoint-initdb.d/` exactly ONCE when `$PGDATA` is empty (first boot). Re-runs are silently skipped — operationally idempotent.

```bash
#!/usr/bin/env bash
# data-plane/postgres/init.sh
# Runs ONCE on first postgres boot when $PGDATA is empty.
# Creates the 3 service users + 3 databases for IntelliFIM v2 Postgres.
set -e

psql -v ON_ERROR_STOP=1 --username "$POSTGRES_USER" --dbname "$POSTGRES_DB" <<-EOSQL
    CREATE USER auth         WITH PASSWORD '${POSTGRES_AUTH_PASSWORD}';
    CREATE USER orchestrator WITH PASSWORD '${POSTGRES_ORCH_PASSWORD}';
    CREATE USER reporting    WITH PASSWORD '${POSTGRES_REPORTING_PASSWORD}';

    CREATE DATABASE auth_backend OWNER auth;
    CREATE DATABASE orchestrator OWNER orchestrator;
    CREATE DATABASE reporting    OWNER reporting;
EOSQL
```

The heredoc uses shell interpolation (`${POSTGRES_X_PASSWORD}`), which is why the file is `.sh` not `.sql`. Standard postgres-image idiom (documented in the official image README under "Initialization scripts").

### 4.3 Extended `init-secrets.sh`

Append to `data-plane/scripts/init-secrets.sh`:

```bash
for var in POSTGRES_ROOT_PASSWORD POSTGRES_AUTH_PASSWORD POSTGRES_ORCH_PASSWORD POSTGRES_REPORTING_PASSWORD; do
    if grep -q "^${var}=.\+" "$ENV_FILE"; then
        continue   # already set, skip
    fi
    secret=$(openssl rand -base64 24 | tr -d '\n')
    if grep -q "^${var}=$" "$ENV_FILE"; then
        sed -i "s|^${var}=$|${var}=${secret}|" "$ENV_FILE"
    else
        echo "${var}=${secret}" >> "$ENV_FILE"
    fi
    echo "${var} written to ${ENV_FILE}"
done
```

Idempotent (same pattern as the existing `JWT_SECRET` generator).

### 4.4 `.env.dataplane.example`

Add 4 new blank lines:

```
POSTGRES_ROOT_PASSWORD=
POSTGRES_AUTH_PASSWORD=
POSTGRES_ORCH_PASSWORD=
POSTGRES_REPORTING_PASSWORD=
```

### 4.5 Fresh-checkout bootstrap order

1. `cp .env.dataplane.example .env.dataplane`
2. `./scripts/init-secrets.sh` → writes JWT_SECRET + 4 Postgres passwords.
3. `docker compose --env-file .env.dataplane up -d` →
   - Postgres entrypoint sees empty `$PGDATA` → runs `init.sh` → creates 3 users + 3 dbs → marks `$PGDATA` initialized.
   - Healthcheck flips to healthy (`pg_isready`).
4. auth-backend / orchestrator / reporting (all `depends_on: postgres: service_healthy`) start after Postgres is ready.
5. Each service `await store.init_schema()` at startup → idempotent `CREATE TABLE IF NOT EXISTS ...`.
6. Each service starts serving traffic.

## 5. Per-Service Store Refactor

The 3 stores' external contracts don't change — same dataclasses, same method signatures, same idempotent `init_schema()` + `aclose()` lifecycle. Only the transport changes.

### 5.1 Common pattern (all 3 stores)

**Config:** each service's `Config` gains a `database_url: str` field parsed from `DATABASE_URL` env var. Fail-fast if missing (same shape as the existing `JWT_SECRET` required-env check).

**Pool, not single connection:** replaces v1's `aiosqlite.Connection` + `asyncio.Lock` with `asyncpg.create_pool(minsize=1, maxsize=8)`. Postgres handles concurrent writes natively — no application-level lock needed.

**Lifecycle:**
- `init_schema()` — if `self._pool is None`: create the pool; acquire one connection; run `CREATE TABLE IF NOT EXISTS ...` and indexes; release. Idempotent.
- `aclose()` — `await self._pool.close()`; `self._pool = None`. Idempotent.
- Every method does `async with self._pool.acquire() as conn:` for its query.

**Placeholder translation (`?` → `$N`):**
```python
# v1 (aiosqlite):
await self._conn.execute(
    "INSERT INTO users(id, email, password_hash) VALUES(?, ?, ?)",
    (uid, email, pw_hash),
)
# v2 (asyncpg):
async with self._pool.acquire() as conn:
    await conn.execute(
        "INSERT INTO users(id, email, password_hash) VALUES($1, $2, $3)",
        uid, email, pw_hash,
    )
```

**Type translation:** Postgres has native `UUID` and `TIMESTAMPTZ` types — asyncpg auto-converts to/from Python `uuid.UUID` and `datetime`. Removes:
- v1's `str(uid)` / `UUID(record["id"])` boundary marshaling.
- v1's `_to_utc_iso(dt)` helper in `reporting/store.py`.
- v1's `datetime.fromisoformat(...)` parse on the read side in `reporting/api.py` (the existing API call site that parses approval `created_at` strings — STAYS because that's parsing HTTP body from the orchestrator, not store data).

**Row decoder:** asyncpg returns `Record` objects (dict-like). The existing `_row_to_X()` helpers stay nearly identical — `record["id"]` access pattern is the same. Just drop the `str → UUID` conversion since `record["id"]` is already a `uuid.UUID` instance.

**`RETURNING` clause:** the orchestrator's "create approval + return the row" pattern can become a single `INSERT ... RETURNING *` query instead of separate INSERT + SELECT. Not required for the migration; can be a Task N+1 cleanup.

### 5.2 auth-backend — `users` table

Postgres DDL inside `init_schema()`:
```sql
CREATE TABLE IF NOT EXISTS users (
    id            UUID PRIMARY KEY,
    username      TEXT NOT NULL UNIQUE,
    email         TEXT NOT NULL UNIQUE,
    password_hash TEXT NOT NULL,
    role          TEXT NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL,
    last_login_at TIMESTAMPTZ
);
CREATE INDEX IF NOT EXISTS idx_users_email ON users(email);
```

Field changes from v1 SQLite: `id TEXT` → `id UUID`; `created_at TEXT` → `created_at TIMESTAMPTZ`; `last_login_at TEXT` → `last_login_at TIMESTAMPTZ`.

The admin-user seed at first boot keeps the same shape — `await store.create_user(username=..., email=..., password=..., role="admin", now=...)`. The `now` parameter goes in directly as a `datetime`, no string conversion.

### 5.3 orchestrator — `approvals` table

```sql
CREATE TABLE IF NOT EXISTS approvals (
    id            UUID PRIMARY KEY,
    host_id       TEXT NOT NULL,
    priority      TEXT NOT NULL,
    score         DOUBLE PRECISION NOT NULL,
    last_reason   TEXT NOT NULL,
    state         TEXT NOT NULL,
    created_at    TIMESTAMPTZ NOT NULL,
    decided_at    TIMESTAMPTZ,
    executed_at   TIMESTAMPTZ,
    decided_by    TEXT,
    error_message TEXT
);
CREATE INDEX IF NOT EXISTS idx_approvals_host_pending
    ON approvals(host_id) WHERE state = 'PENDING';
```

The partial index (per-host PENDING singleton guarantee) — Postgres supports it identically. v1's `UPDATE … WHERE id = ? AND state = ?` + rowcount-check race detection stays, OR can optionally be replaced with `SELECT … FOR UPDATE` inside a transaction (Postgres-native row lock). Either works; recommend keeping the WHERE-clause approach to minimize churn.

### 5.4 reporting — `threat_scores` + `reports` tables

```sql
CREATE TABLE IF NOT EXISTS threat_scores (
    id      BIGSERIAL PRIMARY KEY,
    host_id TEXT NOT NULL,
    score   DOUBLE PRECISION NOT NULL,
    reason  TEXT NOT NULL,
    ts      TIMESTAMPTZ NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_threat_scores_ts ON threat_scores(ts);
CREATE INDEX IF NOT EXISTS idx_threat_scores_host_ts ON threat_scores(host_id, ts);

CREATE TABLE IF NOT EXISTS reports (
    id              UUID PRIMARY KEY,
    name            TEXT NOT NULL,
    range_start     TIMESTAMPTZ NOT NULL,
    range_end       TIMESTAMPTZ NOT NULL,
    generated_at    TIMESTAMPTZ NOT NULL,
    generated_by    TEXT NOT NULL,
    pdf_path        TEXT NOT NULL,
    size_bytes      BIGINT NOT NULL,
    approvals_count INTEGER NOT NULL,
    scores_count    INTEGER NOT NULL
);
CREATE INDEX IF NOT EXISTS idx_reports_generated_at ON reports(generated_at DESC);
```

`AUTOINCREMENT` → `BIGSERIAL` (Postgres-native auto-incrementing 64-bit integer). `size_bytes` becomes `BIGINT` (PDFs can theoretically exceed 2GB; v1 was INTEGER).

`_to_utc_iso(dt)` helper is REMOVED — asyncpg handles `TIMESTAMPTZ` natively.

PDF files stay on the `reporting_data` volume at `/data/reports/{date}-{uuid}.pdf`. Only the metadata table moves to Postgres.

## 8. Testing

### 8.1 testcontainers-python pattern

Each service's `[dev]` extras gain:
```toml
"testcontainers[postgres]>=4.0,<5",
```

Each service's `tests/conftest.py` gains the standard fixture pair:

```python
import asyncpg
import pytest
import pytest_asyncio
from testcontainers.postgres import PostgresContainer


@pytest.fixture(scope="session")
def pg_container():
    """Spin up a Postgres container once per test session."""
    with PostgresContainer("postgres:16-alpine") as pg:
        yield pg


@pytest_asyncio.fixture
async def pg_pool(pg_container):
    """Per-test pool against a fresh database (random name)."""
    import uuid
    db_name = f"test_{uuid.uuid4().hex[:12]}"
    # testcontainers returns a SQLAlchemy URL by default; strip dialect for asyncpg
    raw = pg_container.get_connection_url(driver=None)
    root_url = raw.replace("postgresql+psycopg2://", "postgresql://")
    # Create per-test database
    root_pool = await asyncpg.create_pool(root_url, min_size=1, max_size=1)
    async with root_pool.acquire() as conn:
        await conn.execute(f'CREATE DATABASE "{db_name}"')
    await root_pool.close()
    # Per-test pool against the new db
    db_url = root_url.rsplit("/", 1)[0] + f"/{db_name}"
    pool = await asyncpg.create_pool(db_url, min_size=1, max_size=4)
    yield pool
    await pool.close()
```

Existing test fixtures that previously took `tmp_path` (or used `:memory:`) get rewritten to use `pg_pool`. Most of these are in conftest already (the `store`, `deps`, `tmp_db_path` patterns) — touching conftest cascades to all tests in the file.

### 8.2 Trade-offs of the test approach

| Aspect | Cost |
|---|---|
| Session-scoped container start | ~5–8s once per service's pytest run |
| Per-test CREATE DATABASE | ~50–200ms per test |
| Total overhead per service | ~10–30s slower than v1 SQLite-in-memory |
| Hermeticity | Strong — every test gets a fresh DB; no cross-test pollution |
| Parallel-safety | Yes — random db names prevent collisions |
| Host requirements | Docker daemon (already required for our stack) |

The slowdown is acceptable given the realism gain — we're testing the real driver against the real Postgres engine, catching dialect issues at test time instead of at integration time.

## 9. Error Handling & Edge Cases

| Scenario | Behavior |
|---|---|
| `init-secrets.sh` run before `.env.dataplane` exists | Already fails fast in v1 with `ERROR: $ENV_FILE not found`. v2 inherits same check. |
| `init-secrets.sh` re-run with already-set passwords | Idempotent — `grep` short-circuits per var. |
| Postgres init.sh run on second boot (after first init succeeded) | Skipped by postgres entrypoint — `$PGDATA` is non-empty. |
| Postgres restart loses connections | asyncpg pools auto-reconnect on next acquire; existing connections in flight raise `ConnectionDoesNotExistError` — services log + bubble up. |
| Service starts before Postgres is healthy | Compose `depends_on: condition: service_healthy` gate prevents this. |
| Wrong DATABASE_URL (typo / wrong password) | `asyncpg.create_pool(...)` raises at `init_schema()` time; service exits non-zero; Compose's restart policy retries. |
| DB migration mid-deployment | Out of v2-1 scope — `CREATE TABLE IF NOT EXISTS` handles the additive case; column changes will need alembic later. |
| Pool exhausted (8 concurrent connections) | asyncpg blocks new acquires until one is released; v1's effective max was 1 (`asyncio.Lock`), so v2 is strictly better. |
| Test container start fails (no Docker on host) | testcontainers raises clearly; pytest skips affected tests with an explicit message. Document `docker daemon required` in each service's README. |
